# 디자인 안전 순차 적용 플레이북

이 문서는 **대규모 디자인 변경을 한 번에 반영하지 않고**, 장애/혼선 없이 단계적으로 배포하기 위한 운영 기준입니다.

## 0) 원칙 (반드시 준수)

1. **한 번에 한 축만 변경**
   - 예: 색상 토큰만, 간격만, 컴포넌트 한 묶음만.
2. **기능 로직과 디자인 변경 분리**
   - 디자인 PR에는 기능 로직 변경을 넣지 않습니다.
3. **롤백 가능한 단위로 커밋/PR 분리**
   - 문제가 생기면 특정 단계만 되돌릴 수 있어야 합니다.
4. **모드별 분리 적용**
   - 모바일 → 태블릿 → 데스크탑 순서로 배포/검증합니다.

---

## 1) 적용 단계

### Phase 1 — 토큰/기초 스타일 정리
- 대상: 색상, 폰트, 라운드, 그림자, spacing 등 전역 규칙.
- 금지: 컴포넌트 구조 변경, 이벤트 핸들러 수정.
- 완료 기준:
  - `npm run build` 성공
  - 주요 화면 3개(메인, 환자현황, 설정) 스모크 확인

### Phase 2 — 공통 컴포넌트(원자/분자) 교체
- 대상: 버튼, 입력창, 배지, 모달 헤더/푸터 등 공통 UI.
- 범위 제한: 한번에 3~5개 컴포넌트 이하.
- 완료 기준:
  - 각 컴포넌트 사용 화면 최소 1개 이상 수동 검증
  - hover/focus/disabled 상태 확인

### Phase 3 — 화면 단위 반영 (모바일 우선)
1. 모바일
2. 태블릿
3. 데스크탑

- 각 단계마다 동일 체크리스트 수행:
  - 레이아웃 깨짐 여부
  - 스크롤/터치/키보드 포커스
  - 다크모드(사용 중이면) 시각 확인

### Phase 4 — 고위험 화면 마지막 반영
- 환자현황 그리드, 배드카드 편집/타이머, 모달 다중 오픈 흐름 등.
- 고위험 화면은 한 PR에 1개 화면만 반영.

---

## 2) PR 작성 규칙

- 제목 예시: `design(phase-2): 공통 버튼/입력 컴포넌트 스타일 반영`
- 본문 필수 항목:
  1. 이번 PR의 **대상 범위(화면/컴포넌트)**
  2. **의도적으로 제외한 범위**
  3. **리스크**와 롤백 방법
  4. 테스트/체크 결과 (`npm run build` 포함)

---

## 3) 배포 전 체크리스트

- [ ] 기능 로직 변경이 디자인 PR에 섞이지 않았는가?
- [ ] 모바일/태블릿/데스크탑 최소 1회씩 확인했는가?
- [ ] 핵심 모달(처방 선택, 설정/수정, 이동) 오픈/닫힘 정상인가?
- [ ] 환자현황 입력(이름/부위/처방/메모) 포커스 이동 정상인가?
- [ ] 빌드 성공 (`npm run build`)

---

## 4) 장애 발생 시 롤백 절차

1. 문제 PR 식별
2. 해당 PR만 revert
3. 재빌드 후 빠른 스모크 (`npm run build`, 핵심 화면 3개)
4. 원인 분석 후 더 작은 단위로 재시도

---

## 5) 권장 실행 순서 (이번 프로젝트 기준)

1. `preset-selector` 계열의 시각 토큰 정렬
2. `patient-log` 셀 공통 패딩/폰트/상태 색상 정렬
3. `bed-card` 헤더/푸터/스텝 컬럼 순서로 적용
4. 마지막에 전역 레이아웃(landscape/portrait) 미세조정

위 순서를 지키면, 현재 서비스의 고위험 상호작용(처방 선택/배드 동기화)에 영향을 최소화하면서 디자인을 안전하게 순차 적용할 수 있습니다.
